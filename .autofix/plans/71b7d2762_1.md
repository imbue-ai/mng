# Fix: Replace tuple dict keys with string keys in GitHubData.pr_by_repo_branch

## What the issue is and where

`GitHubData.pr_by_repo_branch` is typed as `dict[tuple[str, str], PrInfo]`. Pydantic's JSON serialization joins tuple keys into comma-separated strings (e.g., `"org/repo,main"`), and deserialization fails because it cannot parse the string back into a tuple. This means `model_dump_json()` followed by `model_validate_json()` raises a validation error.

File: `libs/mng_kanpan/imbue/mng_kanpan/data_types.py`, line 95.

## Why it is a problem

FrozenModel (Pydantic BaseModel) instances are expected to be serializable. While `GitHubData` is currently only used in-memory, a tuple dict key is an unusual pattern that breaks this expectation and will fail if anyone adds JSON logging, persistence, or debugging that serializes the model.

## Planned fix

Replace `dict[tuple[str, str], PrInfo]` with `dict[str, PrInfo]` where the key is a composite string `"repo_path\0branch"` (using a null byte separator that cannot appear in repo paths or branch names). Provide a helper to construct and parse this key.

Alternatively, use a simpler approach: since this is an internal data structure, we could use a nested dict `dict[str, dict[str, PrInfo]]` (repo_path -> branch -> PrInfo). This is more Pythonic and naturally serializable.

I'll go with the nested dict approach as it's cleaner and more idiomatic.

### Changes needed:

1. **data_types.py**: Change `pr_by_repo_branch: dict[tuple[str, str], PrInfo]` to `pr_by_repo_branch: dict[str, dict[str, PrInfo]]`
2. **fetcher.py**: Update all places that read/write `pr_by_repo_branch`:
   - `fetch_github_data`: Change `pr_by_repo_branch[(repo_path, branch)] = pr` to `pr_by_repo_branch.setdefault(repo_path, {})[branch] = pr`
   - `_lookup_pr`: Change `.get((agent_repo, branch))` to nested dict lookup, and update fallback iteration
3. **fetcher_test.py**: Update all test assertions that use tuple keys to use nested dict access

## Risks and edge cases

- The nested dict is mutable within the frozen model (Pydantic doesn't deep-freeze dicts). This is the same as the current tuple-keyed dict.
- The fallback in `_lookup_pr` needs to iterate nested dicts instead of flat iteration. Slightly different code but same semantics.
- All tests must be updated to use the new key format.
