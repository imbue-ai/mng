# Issue 1: ln -s should use -snf to handle stale symlinks

## What the issue is

In `_apply_work_dir_extra_paths` at line 1545 of `libs/mng/imbue/mng/hosts/host.py`, the symlink creation command uses `ln -s` without the `-n` and `-f` flags.

When `_check_local_symlink_state` detects a symlink that points to the wrong target, it returns `(False, False)` -- this falls through to the `ln -s` call. However, `ln -s` without `-n` follows the existing symlink (if it points to a directory) and creates the new symlink INSIDE that directory, rather than replacing the stale symlink. With `-f`, it would remove the existing file first. With `-n`, it treats a symlink-to-directory as a regular file rather than following it.

## Why it is a problem

If a user runs agent creation twice with the same work directory and a changed `work_dir_extra_paths` source path, the stale symlink is not replaced. Instead, a new symlink is created inside the wrong directory. The user ends up with the wrong symlink still in place.

## Planned fix

Change `ln -s` to `ln -snf` on line 1545. This atomically replaces stale symlinks and handles the case where no target exists (same behavior as `ln -s` in that case).

## Risks and edge cases

- `ln -snf` is POSIX-standard and works on both macOS and Linux.
- The `-f` flag means it will also replace dangling symlinks, which is correct behavior.
- The `-n` flag is critical to prevent following symlinks to directories.
