# Issue 1: _check_symlink_state doesn't verify readlink -f success

## What the issue is

In `libs/mng/imbue/mng/hosts/host.py`, the `_check_symlink_state` method at line 1472 compares the stdout of two `readlink -f` commands without checking whether those commands succeeded. On macOS, `readlink -f` on a broken symlink exits with code 1 and produces partial/incorrect output.

## Why it is a problem

If both `readlink -f` commands fail (e.g., both source and target point to non-existent paths), they could produce matching partial output, causing a broken symlink to be incorrectly treated as "correct." While this scenario is unlikely in practice (the source existence is checked before symlink creation), it represents fragile logic that relies on accidental behavior rather than explicit handling.

## Planned fix

Add success checks to both `readlink -f` commands. If either fails, treat the symlink as incorrect (return `is_correct_symlink=False`). This makes the "replace stale symlink" behavior explicit rather than accidental.

Specifically:
- After each `readlink -f` call, check `result.success`
- If either fails, return `(False, False)` -- the symlink exists but is not correct, and it is a symlink (not a non-symlink target)

## Risks

- None significant. The fix makes existing accidental behavior explicit. A broken symlink will be replaced by `ln -snf`, which is the correct behavior.
