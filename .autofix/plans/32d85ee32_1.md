# Issue 1: _classify_entry misclassifies agents whose repo was never fetched

## What the issue is and where

In `libs/mng_kanpan/imbue/mng_kanpan/tui.py`, line 915, the `_classify_entry` function uses:

```python
if agent_repo is not None and repo_pr_loaded.get(agent_repo) is not True:
    return BoardSection.PRS_FAILED
```

This condition conflates two distinct scenarios:
1. The repo is in `repo_pr_loaded` with value `False` (fetch was attempted and failed)
2. The repo is **not in** `repo_pr_loaded` at all (no fetch was attempted -- e.g., all agents for that repo are modal-hosted with no local cwd)

## Why it is a problem

Modal-hosted agents with a `remote` label will have their repo resolved by `repo_path_from_labels`, but `fetch_github_data` skips repos without a local agent work_dir. So `repo_pr_loaded` won't contain the repo. The `.get()` returns `None`, and `None is not True` evaluates to `True`, causing the agent to be placed in the "PRS_FAILED" section with the "PRs not loaded" suffix, even though no fetch failure occurred.

The same issue exists in `enrich_snapshot_with_github_data` (line 142) and `fetch_board_snapshot` (line 208), but those functions handle it correctly by using `(remote.repo_pr_loaded.get(agent_repo) is True) if agent_repo else False` -- which only generates a `create_pr_url` when the repo was actually fetched. The classification logic should be similarly precise.

## Planned fix

Change `_classify_entry` to only classify agents as `PRS_FAILED` when their repo is explicitly present in `repo_pr_loaded` with value `False` (i.e., fetch was attempted and failed). Repos absent from the dict should fall through to `STILL_COOKING`.

Change line 915 from:
```python
if agent_repo is not None and repo_pr_loaded.get(agent_repo) is not True:
```
to:
```python
if agent_repo is not None and repo_pr_loaded.get(agent_repo) is False:
```

Also need to update the `_carry_forward_pr_data` function's similar check on line 870:
```python
and new.repo_pr_loaded.get(agent_repo) is not True
```
should become:
```python
and new.repo_pr_loaded.get(agent_repo) is not True
```
Actually, this one is fine -- in `_carry_forward_pr_data`, we want to carry forward when the repo either failed or is absent from the new snapshot. The "not True" check is correct here because if a repo was previously loaded but is now missing entirely from the new snapshot, that's also a regression worth carrying forward.

The same logic applies in `_has_regressed_repos` -- checking `new_status.get(repo) is not True` is correct because a previously-loaded repo disappearing from the status dict is also a regression.

Add a test case for the fixed behavior: a modal agent with a remote label but whose repo is absent from `repo_pr_loaded` should be classified as `STILL_COOKING`, not `PRS_FAILED`.

## Risks and edge cases

- Need to make sure that `repo_pr_loaded.get(agent_repo) is False` correctly catches the explicit-failure case (value `False` in the dict) while excluding the not-in-dict case (returns `None`). `None is False` evaluates to `False`, so this is correct.
- The existing test `test_classify_entry_failed_repo_goes_to_prs_failed` sets `repo_pr_loaded={"org/repo": False}`, which will still pass since `False is False` is `True`.
- The `_carry_forward_pr_data` and `_has_regressed_repos` checks using `is not True` are intentionally broader and should not be changed.
