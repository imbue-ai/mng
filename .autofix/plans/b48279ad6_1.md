# Issue 1: Use printf instead of echo for path output in shell existence check

## What the issue is and where

In `libs/mng/imbue/mng/hosts/host.py`, line 1499, the batch source-existence check
shell command uses `echo` to output the relative path string when the path exists:

```python
check_parts.append(f"if [ -e {quoted} ] || [ -L {quoted} ]; then echo {shlex.quote(rel_path_str)}; fi")
```

## Why it is a problem

The POSIX specification leaves `echo` behavior undefined when the first argument
starts with `-`. In practice, many Linux shells (bash, dash) interpret `-n` as
"suppress newline", `-e` as "enable escape interpretation", and `-E` as "disable
escape interpretation". If a user configures a `work_dir_extra_paths` key starting
with one of these prefixes, `echo` would swallow or misinterpret the argument
instead of printing it literally. The result: the path would be missing from the
parsed output on line 1507, causing it to be treated as "does not exist" and
silently skipped.

`printf '%s\n'` is the POSIX-standard portable alternative that prints its argument
literally.

## The planned fix

Change line 1499 to use `printf '%s\n'` instead of `echo`:

```python
check_parts.append(f"if [ -e {quoted} ] || [ -L {quoted} ]; then printf '%s\\n' {shlex.quote(rel_path_str)}; fi")
```

## Risks and edge cases

- `printf` is universally available in POSIX shells, so there is no portability risk.
- The `shlex.quote()` around `rel_path_str` ensures the path is passed as a single
  argument to `printf`, so there is no format-string injection risk.
- This is a one-line change with no behavior change for paths that don't start with
  `-n`/`-e`/`-E`.
