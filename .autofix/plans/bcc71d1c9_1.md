# Fix: test_first_load_pr_success_shows_normal_heading missing column_data

## What the issue is

In `libs/mng_kanpan/imbue/mng_kanpan/tui_test.py`, the test `test_first_load_pr_success_shows_normal_heading` (line 1120) constructs an `AgentBoardEntry` without `column_data`, meaning the entry has empty labels.

The test's intent is to verify that when PRs load successfully, the board shows "no PR yet" (STILL_COOKING section) rather than "PRs not loaded" (PRS_FAILED section). However, since the entry has no remote label, `_classify_entry` returns STILL_COOKING because `repo_path_from_labels({})` is `None` -- it never even checks `repo_pr_loaded`. The test passes for the wrong reason.

## Why it is a problem

If someone introduced a bug where agents with a successful repo load were still classified as PRS_FAILED, this test would not catch it. The test should exercise the code path where `agent_repo is not None` and `repo_pr_loaded.get(agent_repo) is True`, causing the entry to fall through to STILL_COOKING.

## Planned fix

Add `column_data=ColumnData(labels={"remote": "git@github.com:org/repo.git"})` to the entry in `test_first_load_pr_success_shows_normal_heading`, consistent with the pattern used in the adjacent `test_first_load_pr_failure_shows_prs_not_loaded` test.

## Risks

None. The test will still pass since `repo_pr_loaded={"org/repo": True}` already correctly maps to the agent's repo. The change only makes the test more meaningful.
