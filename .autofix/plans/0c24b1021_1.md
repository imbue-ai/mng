# Fix Plan: Ambiguous operator precedence in agent_prs_loaded expressions

## What the issue is and where

In `libs/mng_kanpan/imbue/mng_kanpan/fetcher.py`, lines 142 and 208, the expression:

```python
agent_prs_loaded = remote.repo_pr_loaded.get(agent_repo) is True if agent_repo else False
```

is parsed by Python as:

```python
agent_prs_loaded = remote.repo_pr_loaded.get(agent_repo) is (True if agent_repo else False)
```

due to the ternary expression (`X if C else Y`) having lower precedence than the `is` operator.

## Why it is a problem

The intended reading is:
```python
agent_prs_loaded = (remote.repo_pr_loaded.get(agent_repo) is True) if agent_repo else False
```

The actual parsing happens to produce the same results by coincidence:
- When `agent_repo` is a truthy string: both parsings reduce to `repo_pr_loaded.get(agent_repo) is True`
- When `agent_repo` is None: intended gives `False`, actual gives `repo_pr_loaded.get(None) is False` which is `None is False` = `False`

While the behavior is currently correct, this is misleading to readers, fragile under refactoring, and a footgun for anyone maintaining this code.

## Planned fix

Add explicit parentheses to both occurrences (lines 142 and 208) to make the intended precedence unambiguous:

```python
agent_prs_loaded = (remote.repo_pr_loaded.get(agent_repo) is True) if agent_repo else False
```

## Risks and edge cases

None -- this is a pure clarity fix that makes explicit what was already the effective behavior.
