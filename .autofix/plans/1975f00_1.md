# Issue 1: Missing result.success check after batch existence command

## What and Where
In `libs/mng/imbue/mng/hosts/host.py`, line 1500-1501, the batch source-existence
check command result is used without verifying `result.success`:

```python
result = source_host.execute_command("; ".join(check_parts))
existing_paths = set(result.stdout.strip().split("\n")) if result.stdout.strip() else set()
```

## Why it is a problem
If the command fails (e.g., due to SSH connection issues, shell errors, or permissions),
`result.stdout` would be empty and `existing_paths` would be an empty set. This would
cause all configured paths to be silently skipped with "source path does not exist"
warnings -- a misleading message that hides the actual failure.

While the batch command consists of simple `if [ -e ... ]` tests that rarely fail,
defensive error checking is important, especially since this code is designed to work
with remote hosts over SSH.

## Planned fix
Add a warning log when `result.success` is False, so that if the existence check fails,
the user gets a meaningful diagnostic message instead of misleading "does not exist" warnings.

We should warn rather than raise an error, because:
1. The command is a series of independent tests joined by `;`, so partial failure is
   not catastrophic
2. The worst case is that some paths are skipped, which is a safe fallback
3. Raising would break the entire work_dir creation, which is too aggressive for a
   supplementary feature

## Risks and edge cases
- The warning message should be clear enough to diagnose SSH/shell issues
- Should not change behavior for the normal (success) case
